Browser WASM Use-After-Free Demo

This demo simulates a session-management backend implemented in WASM (C). After a session is freed, the pointer is not cleared. Because WASM's linear memory is never returned to the OS, the bytes remain readable. A new same-size allocation typically reuses the freed block, letting attacker-controlled data be read through the stale pointer.

Expected attack flow

  1. Create a session → token is SECRET_TOKEN_42
  2. Free the session → dangling pointer kept, token still readable (stale bytes)
  3. New allocation with attacker data → allocator reuses the freed block
  4. Read token via dangling pointer → returns attacker-controlled data
